# XBPQ 站点选型指南：怎么判断一个网站适合用 XBPQ 规则来写

> 本文回答一个问题：**拿到任意影视网站，如何判断它该不该用 XBPQ 规则（写一份 JSON）来接入，还是得上 Java / JS？**
> 配套文档：`XBPQ规则引擎分析.md`（引擎原理）、`开发文档.md` 第十节（XBPQ 是什么）。

---

## 一、先理解 XBPQ 的能力边界（一句话）

XBPQ 本质是「**HTTP 取原文 + 字符串边界截取 + URL 模板拼接**」的规则引擎。它能做的全靠"文本切割"完成，不能做的（执行 JS、算签名、跑解密）靠再多规则也补不上。

它**没有 JS 执行引擎**（那是 JS 派 quickjs 的活），只能发 GET / 简单请求 + 截字符串。

---

## 二、四道硬门槛（有一条不过 → 基本不适合）

| # | 判断点 | 适合 ✅ | 不适合 ❌ |
|---|--------|--------|----------|
| 1 | **数据是否在 HTML 源码里** | 右键"查看网页源代码"能直接搜到片名 / 播放地址 | 数据全靠 JS 渲染，源码里只有空 `<div id="app">`（React / Vue SPA） |
| 2 | **列表 URL 是否可模板化** | 分类页地址有规律，如 `/show/{cateId}-{area}---{catePg}.html` | 翻页 / 分类靠 POST 表单、靠 JS 计算参数、靠动态 token |
| 3 | **播放地址获取难度** | 播放页源码能直接截到 `.m3u8 / .mp4`，或固定嗅探即可 | 播放地址要跑 JS 解密、算签名、多次跳换算 key |
| 4 | **反爬强度** | 无验证码、无强制登录、UA / Referer 能满足 | Cloudflare 5 秒盾、滑块、需登录 Cookie、频繁风控 |

> 第 1、3 条最关键，因为它们直接决定 XBPQ 能不能"截到东西"。

---

## 三、按数据形态把网站分三类

### ✅ 强烈适合：苹果CMS（maccms）站群

LIBVIO 样例（`XBPQ规则引擎分析.md` 第四节）就是典型 maccms 模板站。特征：

- URL 高度规律：`/show/分类-地区-排序-类型-----页码---年份.html`
- 详情 / 播放页结构固定（`stui-content`、`stui-vodlist` 这类类名）
- 全服务端渲染，源码里数据齐全

**判断技巧**：源码里搜到 `stui-`、`module-`、`vodlist`、`/vodplay/`、`/detail/` 这类字样，八成是 maccms → 直接套模板改改就能用。全网这类站占比极高，这也是 XBPQ 流行的根本原因。

### 🟡 勉强适合：JSON 接口站

网站有明确 API 返回 JSON（如 `/api/list?cateId=1`），XBPQ 有 `json数组` 路径模式（如 `data.list[1].name`）可解。

- 适合：接口无签名、参数能模板化拼进 URL
- 不适合：接口要 POST body、要 token、要时间戳签名

### ❌ 不适合：SPA / 强 JS 站

- Netflix 式前端渲染、B 站这类要 wbi 签名、需要跑 JS 解密播放源的
- 这类只能用 **jar 派（写 Java，如本项目）** 或 **JS 派（jadehh/TVSpider）**

## 四、5 分钟快速自测流程

拿到一个网站，按顺序做这 5 步，全过 → 用 XBPQ：

1. **看源码**：`Ctrl+U` 查看首页源代码，`Ctrl+F` 搜一个已知片名 —— **搜得到才继续**。
2. **翻分类**：点分类、翻第二页，看 URL 变化规律 —— 能找到 `{分类}{页码}` 规律才继续。
3. **进详情**：看详情页源码里，演员 / 简介 / 线路 / 剧集列表是不是明文 —— 能定位"左边界 `&&` 右边界"才继续。
4. **看播放**：进播放页看源码，能否截到 `.m3u8 / .mp4`，或用通用嗅探能拿到 —— 能拿到才继续。
5. **测搜索**：搜索是 GET（`/search?wd=xx` 或 `/搜索词/detail/`）还是 POST —— GET 最好，POST 且带签名就麻烦。

---

## 五、判断口诀

> **"源码可见、地址有律、直链好取、没有强盾" → 上 XBPQ；**
> **"渲染靠 JS、参数靠算、播放靠解密" → 别用 XBPQ，写 jar / JS。**

XBPQ 的性价比场景就是：**maccms 类站群批量接入**——同一套模板改几个 CSS 选择器和 URL 就复制一个站，写一份 JSON 十几分钟，这是写 Java 类完全比不了的效率。而一旦遇到需要"计算 / 执行 / 解密"的站，它就无能为力，这正对应它与 jar 派 / JS 派的分工边界。

---

## 附：体检清单（逐项勾选）

- [ ] 首页源代码能搜到已知片名
- [ ] 分类 / 翻页 URL 有规律、可模板化
- [ ] 详情页影片名 / 简介 / 演员为明文，可定位 `&&` 边界
- [ ] 播放页可截到 `.m3u8 / .mp4` 直链
- [ ] 搜索为 GET 请求
- [ ] 无 Cloudflare / 验证码 / 强制登录
- [ ] （可选）源码含 `stui- / vodplay / detail` 等 maccms 特征

**勾满前 6 项 → 强烈推荐 XBPQ；勾中"可选"项 → 直接套 maccms 模板。**
